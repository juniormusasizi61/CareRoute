2. Engineering Practices Framework
Modern software engineering is not just about writing code that works — it is about writing code that is
maintainable, secure, testable, and deployable at any moment. The six practices below are woven into every
phase of this project. Each subsequent section will reference which practice applies at that point in
development.
PRACTICE 1
Robust Version Control & Branching
Trunk-based development: all branches merged within 24–48 hours
Branch naming: feat/, fix/, chore/, docs/, test/ prefixes enforced
Conventional commits: feat(api): add JWT refresh rotation
Protected main branch: PRs required, linear history enforced
Git hooks (Husky): run lint + type-check before every commit
PRACTICE 2
Streamlined CI/CD Pipelines
Every push to a feature branch triggers: lint → type-check → unit tests
Merges to main trigger: full test suite → Docker build → staging deploy
Progressive delivery: feature flags gate new route algorithms in production
Rollback procedure documented: git revert + re-deploy < 5 minutes
Deployment environments: local → dev → staging → production
PRACTICE 3
Comprehensive Automated Testing
Testing pyramid: 70% unit, 20% integration, 10% E2E
Coverage targets: 80%+ on optimiser service; 75%+ on API routes
Contract tests: Pact.io validates API ↔ optimiser interface
Playwright E2E: critical dispatcher flow tested on every PR
Mutation testing (Stryker): validates test suite quality
PRACTICE 4
Code Reviews & Peer Collaboration
All PRs require at least one approval before merge
PR template enforces: summary, test plan, screenshot/recording for UI
ESLint + Prettier auto-enforce style; reviewers focus on logic only
Architecture Decision Records (ADRs) document major decisions
Weekly sync: rotating ownership of complex modules

3.2  Database Bootstrap with PostGIS
PostGIS extends PostgreSQL with spatial data types and functions. The GEOGRAPHY type stores real-world
coordinates with automatic geodesic distance calculations. Always use GEOGRAPHY over GEOMETRY for
lat/lng data unless you are working in a projected coordinate system.
-- Enable PostGIS (run once after CREATE DATABASE)
CREATE EXTENSION IF NOT EXISTS postgis;
CREATE EXTENSION IF NOT EXISTS postgis_topology;
CREATE EXTENSION IF NOT EXISTS pg_trgm; -- for fuzzy address search
-- Verify

3.3  JWT Authentication System
CareRoute uses a dual-token strategy. A short-lived access token (15 minutes) is sent in the Authorization
header. A long-lived refresh token (7 days) is stored in an HTTP-only Secure cookie. On expiry the frontend
silently exchanges the refresh token for a new pair. Refresh token IDs are stored in Redis with TTL; logging
out adds the token to a Redis blocklist, making it immediately invalid.

3.4  User & Worker Management API
The API exposes role-based REST endpoints. Dispatchers manage workers; workers can only read their own
assigned routes and post status updates. Middleware validates JWT and attaches the decoded user context
to every request.

POST /api/v1/auth/logout - Revoke refresh token
GET /api/v1/workers - List workers (DISPATCHER only)

4.1  PostGIS Schema for Locations & Routes
The geospatial schema stores three types of geometry: point locations for addresses and worker home
bases, linestrings for route paths between stops, and polygons for service zone boundaries. All geometry is
stored as GEOGRAPHY(type, 4326) — WGS84 coordinate system — ensuring ST_Distance returns metres
without a projection transformation.
-- apps/api/prisma/migrations/002_spatial.sql
CREATE TABLE stops (
id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
job_id UUID NOT NULL REFERENCES jobs(id) ON DELETE CASCADE,
address_raw TEXT NOT NULL,
address_norm TEXT,
location GEOGRAPHY(POINT, 4326) NOT NULL,
sequence INTEGER, -- assigned by optimiser
worker_id UUID REFERENCES workers(id),
status stop_status NOT NULL DEFAULT 'PENDING',
time_window_start TIMESTAMPTZ, -- earliest allowed arrival
time_window_end TIMESTAMPTZ, -- latest allowed arrival
service_duration INTEGER DEFAULT 900, -- seconds on-site
completed_at TIMESTAMPTZ,
created_at TIMESTAMPTZ DEFAULT NOW()
);
CREATE TYPE stop_status AS ENUM ('PENDING','IN_PROGRESS','COMPLETED','SKIPPED');
-- GiST spatial index (essential for performance)
CREATE INDEX idx_stops_location ON stops USING GIST(location);
CREATE INDEX idx_stops_job_worker ON stops(job_id, worker_id);
-- Route path geometry
CREATE TABLE route_segments (
id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
route_id UUID NOT NULL REFERENCES routes(id),
path GEOGRAPHY(LINESTRING, 4326) NOT NULL,
from_stop UUID REFERENCES stops(id),
to_stop UUID REFERENCES stops(id),
distance_m NUMERIC(10,2),
duration_s INTEGER
);

4.2  Mapbox GL JS Integration (Frontend)
The dispatcher dashboard uses react-map-gl (Mapbox GL JS wrapper for React) to render an interactive
route map. Each worker's stops are drawn as a coloured GeoJSON layer. Route paths from the
route_segments table are drawn as LineString layers. Clicking a stop opens a popup with job details and a
status update button.
// apps/web/src/components/RouteMap.tsx
import Map, { Source, Layer, Popup, Marker } from 'react-map-gl';
import { useRouteStore } from '@/store/routes';
export function RouteMap() {
const { routes, selectedStop, setSelectedStop } = useRouteStore();
// Build GeoJSON from routes data
const routeGeoJSON = buildRouteGeoJSON(routes);
return (
<Map
mapboxAccessToken={import.meta.env.VITE_MAPBOX_TOKEN}
mapStyle="mapbox://styles/mapbox/streets-v12"
initialViewState={{ longitude: 36.82, latitude: -1.29, zoom: 11 }}
>

<Layer id="route-lines" type="line"

</Map>
);
}
